# Owner portal — QA findings (as-a-user walkthrough)

Generated by driving the app like a real user (Detox), screenshotting every
screen + post-action state, and reviewing each screenshot for what a paying
user would actually see. This is the user-experience layer on top of the
31-screen render coverage + the create-project functional flow.

## Method
- **Tour:** drove all 31 owner screens, captured a screenshot of each, fanned
  out 7 agents to review them as a user.
- **Every-button pass:** taps all 166 stay-on-screen safe controls across 19
  screens, screenshotting each, reviewed for dead/broken buttons.
- **Functional flows:** create-project (form → DB), edit business info, estimate
  builder.

---

## FIXED (this pass)
- **"1 invoices overdue" → "1 invoice overdue"** — i18next plural keys added
  (`overdueInvoices_one`/`_other`) for en, pt-BR, es.
- **Clients screen showed $0 / 0 projects for every client** — root cause: the
  seeded projects stored `client_name` as text but were never linked to the
  `clients` table via `project_clients`, which the backend aggregation
  (`GET /api/portal-admin/clients`) requires. Seeded the 4 missing links;
  the endpoint now returns correct per-client revenue/project counts.
  ⚠️ **PRODUCT QUESTION for review:** does the create-project form ever create
  a `project_clients` link? The in-app create-flow only sets `client_name` text,
  so real owners may also see $0 client stats unless linking happens elsewhere.
- **estimatesDetail showed `$0 / Unnamed Client` rows** — these were
  test-pollution blank drafts auto-created by repeatedly opening the estimate
  builder. Deleted; the 4 real estimates ($85k/$20k/$15k/$9.5k) remain.
  ⚠️ Test hygiene: opening the estimate builder auto-creates a draft — the
  tour/flow specs should clean up after themselves.

## NEEDS YOUR CALL (documented, not changed — risk/ambiguity)
- **Currency formatting shows `$56.000` (period thousands) for an `en` owner.**
  `formatCurrency` is correct (returns en-US for `language='en'`), so the app
  appears to be running in **pt-BR locale despite the owner profile being `en`**
  — i.e. `i18n.language` isn't following `profile.language` (or the sim's device
  locale is winning). Also `ProjectDetailView` uses 10 raw `.toLocaleString()`
  calls (device locale, ignores app locale entirely). Decision needed: force US
  format for USD? follow profile language? fix the en→pt-BR sync? Then a
  consistent fix across screens.
- **projectDetail: "Income $12,000 / 100% collected" vs Billing "Collected
  $3,200 / Outstanding $8,800".** Two different concepts (manual income txns vs
  invoice draws) shown without reconciliation — confusing to a user. Needs a
  product decision on labeling/derivation.

## Every-button pass (166 buttons driven, 101 reviewed by agents)
- **50 buttons directly verified WORKING** from screenshots: estimate/invoice
  list filters (all/sent/draft/accepted/rejected) apply correctly, section
  accordions expand, financial report Company/By-Project toggle switches the
  view, contracts upload opens the source picker, search clear resets, etc.
- **No genuinely dead buttons found** this pass. The 36 "dead" review verdicts
  were all traced to TEST ARTIFACTS, not product bugs:
  - **Tab-state pollution:** tapping `work.servicesTab` switches to the (empty)
    Services segment, which persists in AsyncStorage — so later Projects-based
    screens loaded on "No Service Plans Yet", making `ownerProjects` filters look
    dead and `projectDetail`/`projectDocuments` nav fail. Harness should reset
    the segment between screens.
  - **Lingering native alert:** the "Extract from photo" action sheet stayed
    open and blocked subsequent estimateBuilder taps (the no-relaunch speed
    trade-off). Spec should dismiss alerts between buttons.
  - **Modal-not-open / precondition:** `ownerWorkers` payment-type options live
    inside an Add-Worker modal; `ownerDashboard.addWidgetButton` needs edit mode;
    `helpSupportItem` opens an external mail app Detox can't capture.
  - Spot-checked the plausibly-dead ones in code — all wired
    (`setShowAddSheet`, `toggleTheme`, `chooseEmailApp`, filter handlers).
- **Real dead button already fixed earlier:** BillingCard chevron (rendered
  tappable on non-tappable rows).
