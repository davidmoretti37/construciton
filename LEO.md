## Estimate Creation Flow ‚Äî Full Trace

```
User says "create an estimate" in chat
  ‚Üí CoreAgent routes to EstimateInvoiceAgent
  ‚Üí AI generates JSON with estimate-preview visual element
  ‚Üí ChatScreen renders EstimatePreview card with Save/Share/Preview buttons
  ‚Üí User taps Save
  ‚Üí handleSaveEstimate() in useEstimateActions.js
  ‚Üí Resolves project_id (3 fallback strategies)
  ‚Üí Calls saveEstimate() from utils/storage/estimates.js
  ‚Üí Inserts to estimates table, optionally updates linked project
```

## FAILS (build blockers)

### FAIL-1: Flat `tasks` array never persisted
**Where:** AI prompt generates `tasks: [{description: "Demo existing bathroom"}, ...]` (estimateInvoicePrompt.js:462-469). EstimatePreview.js:745-757 renders them. `saveEstimate()` at estimates.js:37 saves `phases: estimateData.phases || []` but there is no `tasks` column and `tasks` is never mapped into any JSONB field.  
**Impact:** User sees task checklist in preview. After save+reload, zero tasks. 100% data loss on every flat estimate.  
**Fix:** Store tasks inside the `scope` JSONB column: `scope: { ...estimateData.scope, tasks: estimateData.tasks || [] }` in `saveEstimate()`.

### FAIL-2: `laborEstimate` never persisted
**Where:** AI generates `laborEstimate: {workersNeeded, daysNeeded, laborCost, reasoning}` (prompt line 425-435). EstimatePreview.js:48,66-69,868-919 renders it with cost calculation. `saveEstimate()` never writes it anywhere.  
**Impact:** Labor planning data (workers x days x rate) visible in preview, gone after save. Owner loses labor cost awareness on reload.  
**Fix:** Store in scope JSONB: `scope: { ...estimateData.scope, laborEstimate: estimateData.laborEstimate }`.

### FAIL-3: Estimate `date` field never persisted
**Where:** AI generates `date: "2026-04-17"` (prompt line 458). For project-linked estimates, prompt line 181 instructs AI to use the project START DATE. EstimatePreview.js:39,613 renders it. No `date` or `estimate_date` column exists.  
**Impact:** After save, only `created_at` (always "now") survives. An estimate dated to a future project start loses its date. PDF generation may show wrong date on reload.  
**Fix:** Add `estimate_date DATE` column to estimates table; persist in `saveEstimate()`.

### FAIL-4: Client city/state/zip never persisted
**Where:** AI generates `clientCity`, `clientState`, `clientZip` as separate fields (prompt lines 453-456). EstimatePreview.js:33-36,81-89 renders the full formatted address. `saveEstimate()` at estimates.js:33 saves only `client_address: estimateData.clientAddress` ‚Äî just the street line.  
**Impact:** Full "123 Main St / Miami, FL 33101" shows in first preview. After save+reload, only "123 Main St" remains. PDF "Bill To" and "Ship To" sections show incomplete addresses.  
**Fix:** Concatenate on save: `client_address: [clientAddress, [clientCity, clientState].filter(Boolean).join(', '), clientZip].filter(Boolean).join(', ')`. Or add separate columns.

### FAIL-5: `createInvoiceFromEstimate()` drops `project_id`
**Where:** estimates.js:630 ‚Äî SELECT query fetches `id, client_name, client_phone, ...` but does NOT include `project_id`. Lines 645-661 ‚Äî INSERT into invoices never sets `project_id`.  
**Impact:** "Convert to Invoice" button (EstimatePreview.js:492-495) creates an invoice orphaned from the project. Invoice won't appear in project financial reports. The invoices table HAS a `project_id` column (migration line 81) ‚Äî it's just never populated via this path.  
**Fix:** Add `project_id` to the SELECT on line 630; add `project_id: estimate.project_id` to the INSERT on line 646. Two-line fix.

## FLAGS (fixable issues)

### FLAG-1: Invoice pricing intelligence uses wrong field name
**Where:** invoices.js:58 references `item.pricePerUnit` for pricing history recording. AI prompt (line 471) generates items with `price` field, not `pricePerUnit`.  
**Impact:** Every invoice's pricing intelligence records `undefined` as unit price. AI pricing learning from invoices is completely broken.  
**Fix:** `pricePerUnit: item.pricePerUnit || item.price` at invoices.js:58.

### FLAG-2: `saveEstimate` retry path skips input validation
**Where:** estimates.js:56-86 ‚Äî retry on duplicate estimate_number uses raw `subtotal: estimateData.subtotal || 0` instead of `Math.max(0, parseFloat(...))` guards used in primary insert (lines 39-42).  
**Impact:** If retry fires, negative values could be inserted. Low probability but inconsistent safety.  
**Fix:** Extract insert payload to shared variable.

### FLAG-3: `handleSaveEstimate` dead code for flat estimates
**Where:** useEstimateActions.js:56-57 checks `estimateData.phases?.some(p => p.tasks?.length > 0)`. For flat estimates (no phases, just tasks+items), this is always false, triggering a message-scanning loop (lines 59-85) that never finds phases either.  
**Impact:** Wasted computation, no data corruption. The fallback to conversation state (line 89) and name search (line 98) still correctly resolve project_id.  
**Fix:** Add check: `const hasTasks = estimateData.tasks?.length > 0 || actionHasTasks`.

### FLAG-4: `profit` field not persisted
**Where:** EstimatePreview.js:117-119 calculates profit margin. The preview shows profit breakdown (lines 835-850). No `profit` column in estimates table.  
**Impact:** If AI generates an estimate with profit markup, it displays correctly but is lost on save. Currently the AI prompt doesn't explicitly generate profit, so this is latent.  
**Fix:** Store in scope JSONB or add column when feature is actively used.

## Strengths

1. **Robust project_id resolution** ‚Äî `handleSaveEstimate` has 3 fallback strategies: direct UUID ‚Üí conversation state ‚Üí name search. Well-engineered for AI output variability.

2. **Pricing intelligence feedback loop** ‚Äî Records every estimate's pricing and tracks owner corrections with 1.5x weight. Self-improving pricing engine.

3. **Full inline editing** ‚Äî EstimatePreview has live total recalculation, line item add/remove, date picker, labor estimate editing, and address editing. Comprehensive edit-in-place UX.

4. **Estimate-to-project sync** ‚Äî When saving a linked estimate, auto-updates project budget, contract_amount, base_contract, dates, scope, and phases. Good bidirectional flow.

5. **Multi-format field handling** ‚Äî Both prompt and storage handle `client` as string or object, `clientPhone`/`client_phone`, `project_id`/`projectId`. Defensive against AI variability.

6. **Owner-supervisor aggregation** ‚Äî `fetchEstimatesForOwner` and `fetchInvoicesForOwner` properly aggregate across hierarchy with attribution.

---

## Final Verdict

**APPROVE WITH CHANGES** ‚Äî The estimate creation flow works well for the initial preview experience, but has a systematic problem: **the DB schema hasn't kept up with the AI output format**. Five fields generated by the AI (tasks, laborEstimate, date, city/state/zip, profit) are displayed in the preview but silently lost when saved. The most critical fix is FAIL-5 (invoice conversion dropping project_id) because it breaks financial tracking ‚Äî an invoice without a project link means the project's P&L is wrong.

Priority order for fixes:
1. **FAIL-5** ‚Äî Add `project_id` to invoice conversion (2-line fix)
2. **FAIL-4** ‚Äî Persist full address (concatenate or add columns)
3. **FAIL-1** ‚Äî Persist tasks array (store in scope JSONB)
4. **FAIL-3** ‚Äî Persist estimate date (add column)
5. **FAIL-2** ‚Äî Persist labor estimate (store in scope JSONB)
6. **FLAG-1** ‚Äî Fix pricing intelligence field name mismatch

None of these are architectural changes ‚Äî they're all column additions or field mappings in the save functions. Total estimated effort: 1-2 hours.
