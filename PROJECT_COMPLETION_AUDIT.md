# Sylk — Project-Based Completion Audit

> Generated 2026-06-16 by a 106-agent multi-agent audit (18 subsystem auditors → adversarial verification → completeness critic → synthesis). Every finding is grounded in the current code with file:line evidence.

**Overall completeness for a project-based product: 63%**

## Executive Summary

Sylk can run the MIDDLE of a project-based job today — projects/phases/tasks, scheduling, estimates, change orders, progress (draw) billing with retainage, Stripe payments, expenses/financial reports, bank connect + manual reconciliation, crew time clock, daily reports/photos, and a notably complete subcontractor subsystem, all drivable by a solid, well-unit-tested Foreman agent. The honest overall completeness for a PROJECT-BASED product is about 63%. The gating problems are not depth features; they are confirmed breaks in the headline money loop and in monetization. The customer signing page /sign/[token] does not exist (verified), signing emails are never sent, change orders cannot be signed in-app, the InvoiceBuilder Send button is dead and edits lock read-only after the first save, manual/check/ACH payments never record as income (verified), the bank auto-match RPC is dead due to a non-existent column reference (verified), and a one-line missing import throws in financial.js (verified). Worse for the business: the paywall is hardcoded off and bypassable via direct frontend inserts — the product charges nobody. The lifecycle also has no head and no tail: there is no lead intake or sales pipeline, estimates require a pre-existing project, customer is not a first-class entity, and there is no closeout/warranty flow or retainage release (held money has no exit) — yet the pivot is explicitly 'lead -> ... -> closeout'. The pragmatic path is: Phase 0 fixes the sign/pay loop and turns on enforced monetization (mostly S/M fixes to already-built features); Phase 1 stitches the estimate->draw->invoice->retainage pipeline and adds the lead/customer/closeout ends; Phase 2 makes the product importable, multi-user, and tested; Phase 3 adds field-offline, permits/inspections, dependency scheduling, and reporting depth. Separately, the route/service-plan de-scoping is itself a real completion RISK because it made core tables nullable and entangled clock-out/expense/reconcile — it must be unwound surgically, not deleted, and after removal the audit log will be nearly empty of project events.

## State of the Core

The MIDDLE of a project genuinely works today. An owner can create a project with phases/sections and tasks (75% — works end-to-end, though creation is non-atomic and task data is dual-stored in project_phases.tasks JSONB AND worker_tasks); auto-schedule tasks across phases with tested distribution math; build estimates and change orders; run progress (draw) billing that correctly holds retainage; take Stripe card payments end-to-end; record expenses and produce P&L/AR-aging/cash-flow/tax reports; connect a bank via Teller and manually match transactions; clock crews in/out and compute gross payroll; capture daily reports and photos; and the subcontractor subsystem (bids->engagements->compliance->sub portal->sub payments, 80%) is the most complete part of the app. The Foreman agent core (82%) is solid and well-tested at the unit level, with strong tool def<->handler parity and a working approval gate, and it can drive most of these operations via chat. Estimates, change orders, invoices, and the e-sign PDF/hash pipeline each function in isolation. So the product can convincingly demo a job that is already underway. What it CANNOT yet do is capture the job at the start (no leads/pipeline, no first-class customer, estimates require a pre-existing project), reliably get documents signed or invoices delivered/paid (the /sign web page does not exist, signing emails are never sent, COs have no client signing surface, the InvoiceBuilder Send button is dead, manual payments never book as income), or finish the job (no closeout/warranty, retainage is held forever with no release). The bank auto-match — a headline value prop — is dead due to a column-name bug, and crucially the product charges nobody because the paywall is hardcoded off and bypassable. Test coverage is strong on agent plumbing and pure calculators but nearly absent on the financial-correctness and lifecycle-write handlers, which is exactly where the confirmed runtime bugs live.

## 🗺️ Completion Roadmap

### Phase 0 — Stop-the-bleeding: make the money loop and monetization actually function (1-2 wks)

*A credible end-to-end demo where a document gets signed, an invoice gets delivered and paid (incl. check/cash), payments book as income, and the product can charge. These are confirmed breaks in headline features, mostly S/M effort.*

- Ship the Next.js /sign/[token] page (render PDF + signature canvas, POST /api/esign/sign/:token) and enable sendEmail at every createSignatureRequest caller (changeOrders/portalOwner/engagements/documents).
- Add /change-orders/:id/signing-link endpoint + in-app CO signing (WebView or SignaturePad mirroring SignDocumentScreen) so signature-required COs are approvable from the portal.
- Re-enable the contract e-sign UI in ContractsScreen.js (uncomment request button + SignatureSection) now that /sign exists.
- Fix InvoiceBuilder: introduce a real 'draft' status (or change the readOnly gate), bind handleSend() to a visible Send button.
- Fix financial.js: import resolveProjectId (one-line) — restores assign_bank_transaction and project-scoped P&L.
- Record income on manual/check/cash/ACH payment: insert an income project_transaction (or a DB trigger) from update_invoice when amount_paid increases.
- Fix the bank auto-match RPC: correct the project_transactions.user_id reference (it has no user_id) and add a functional test for the >=0.85 branch.
- Turn off TESTING_MODE, wire useSubscriptionBlock into navigation, route project creation through a backend endpoint that enforces can_create_project, and add a checkout gate to onboarding.
- Fix daily-report photo upload path to satisfy documents-bucket RLS (or switch to the project-photos bucket) so field photos persist on a correctly-configured prod.

### Phase 1 — Close the lifecycle ends + stitch the seams (2-4 wks)

*The product can run lead->...->closeout end-to-end, and the estimate->draw->invoice->retainage money pipeline is stitched rather than islands.*

- Lead intake + pipeline: add leads table, pipeline stages, lost-reason, a lead screen, and allow estimating a prospect without a pre-existing project (relax portal.js:725 / add lead-scoped estimate).
- Estimate-accept hand-off: on accept, advance project, seed draw schedule from the accepted estimate's line items, create deposit invoice, and trigger schedule.
- Retainage release / final-billing: a path that issues a final invoice releasing accumulated retainage_held.
- Project closeout flow: punch-list entity, completion workflow, warranty start/expiry, completion certificate + final lien waiver in a closeout package surfaced to the client.
- First-class Customer entity: customer table (can exist pre-project), customer_id FK on projects/estimates/invoices, create/edit/detail UI; fix the multi-project portal by reading customer's projects with a switcher.
- Fix the change-order audit trigger (entity_id/company_id) or surface approval_events directly, add an owner CO detail screen with approval history + signed-PDF link.
- Phase-tagged job cost: add phase_id on clock-out labor and on bank/expense assignment, add a phase-level spend rollup; deliver estimated-vs-actual budget-vs-actual by phase.
- Worker payment ledger: worker_payments/payroll_run table with mark-paid/pay-date/method; fix get_payroll_summary's bad columns and the PayrollSummaryScreen clock_in_records reference.
- Server-rendered invoice PDF (attach to email + store pdf_url + portal download) and a send_invoice Foreman tool with payment link.
- Capture missing migrations (oauth_connections, projects.drive_folder_id, sms_messages + profiles.twilio_*) so fresh deploys work.

### Phase 2 — Make it multi-user, importable, and reliable (3-6 wks)

*An established contractor can migrate in, the team is genuinely collaborative, and the data-destructive paths are transactional and tested.*

- **Lead intake / sales pipeline BEFORE the estimate (lead capture form, web/inbound lead, lead -> qualified -> estimate -> won/lost stages, lost-reason).**

### Cross-subsystem gaps

- ESTIMATE -> PROGRESS-BILLING -> AR loop has a hole at BOTH ends. Front end: accepting an estimate (portal.js:703) does not generate the draw schedule or a deposit invoice -- the estimate's line items/totals never seed the draw schedule (create_draw_schedule takes manually-typed items, not the accepted estimate). The 'auditability' seam spans Auth + Change Orders + the dropped modes and currently produces an empty, misleading trail for the project lifecycle.

## CRM Customer Model & Power-User Horizontals

**Completeness:** 18% · **Works end-to-end:** no · **Tests:** None found for this subsystem. No test references portal-admin/clients, /share, the clients enrichment endpoint, ClientsScreen, or the audit-customer wiring. The only adjacent tests are servicePlanTools.test.js, which covers the out-of-scope service-plan customer data.

> Bottom line: there is no real CRM customer model for project-based work. A 'client' is a portal-login account created as a side effect of sharing a project; it cannot be created standalone, edited, or opened, has no tags/type/status/notes/second-contact/property, and is not linked by FK to estimates or invoices (those carry loose name/email text). The lead/pipeline stage of the stated lifecycle is entirely absent. Of the requested horizontals: activity-feed infra exists (audit_log + /api/audit + AuditTrail component) but the customer feed is unwired (zero customer audit writes); project/contract/invoice TEMPLATES exist; tags, custom fields, saved views, bulk actions, and comments/@mentions do NOT exist anywhere. Site-access notes and multi-contact exist only in the out-of-scope route subsystem and need to be rebuilt on the project/customer entity. Recommended sequencing: (1) promote Customer to a first-class entity with customer_id FKs and a create/edit/detail UI, (2) add lead pipeline, (3) add multi-contact + multi-property + site-access notes, (4) wire customer audit writes so the existing activity feed populates, (5) strip the service_plan_count leak from the clients endpoint/UI.
